## 二、修正后的核心事实

### 2.1 阻塞行为的真正来源

**阻塞语义来自 OMO Slim v1.1.1 的 `subtask` 工具实现**，而非 superpowers 技能指令。

- OMO Slim v1.1.1 的 `subtask` 工具：同步等待子会话完成
- OMO Slim orchestator prompt (L197-198)：明确写 "Delegation is blocking"
- superpowers 技能：只描述 "dispatch subagent, collect result, proceed"，对底层同步/异步不可知

### 2.2 superpowers 技能的实际状态

| 技能 | 状态 | 需要任何改动 |
|------|------|-------------|
| subagent-driven-development | 无 subtask 引用，工具无关 | **不需要** |
| dispatching-parallel-agents | 纯概念性指令 | **不需要** |
| using-superpowers | 零运行时语义 | **不需要** |
| executing-plans | 串行回退技能 | **不需要** |
| verification-before-completion | 纯规则，无工具引用 | **轻量扩展**（加持久化条款） |
| brainstorming | 设计流程 | **不需要** |
| writing-plans | 计划流程 | **不需要** |
| （新增）background-orchestration | 异步编排 | **需要新增** |

### 2.3 结论

**superpowers 技能包完全不需要 fork。** 唯一需要的是一个新增的 companion skill 加上对 verification 的轻量扩展。这从原估计的"fork 整个仓库并重写 4-6 个技能文件（2-4周）"降为"新增 1 个技能文件 + 1 个文件轻量扩展（1-2天）"。

## 四、修正后的推荐路线

### 明确推荐：修正版路径A（补丁移植 + 轻量扩展）

三位 councillor 在具体策略上有分歧（alpha 偏好 A+B 合并、beta 偏好简化版 B、gamma 偏好 A'），但**共识区间完全重叠**：

1. 将 patch-kit 的访问控制补丁移植到 OMO Slim v2.0.0 源码
2. 新增 1 个 companion skill 处理异步回调
3. 轻量扩展 verification-before-completion
4. 不 fork superpowers
5. 工作量 1-1.5 周

### 实施步骤

**第 1 步（1 天）：静态策略文件移植**
- 15 个独立文件（superpowers-policy.ts, agent-tier-policy.ts, agent-mcp-blacklist.ts, orchestrator-only-skills.ts, merge-agent-config.ts, cooldowns.ts 等）零依赖 subtask，直接放入 v2.0.0 对应目录
- Agent factory deny 块追加 `cancel_task: 'deny'`
- MCP 名 `grep_app` → `gh_grep` 全局替换

**第 2 步（1 天）：最小 patch 集成点**
- `src/index.ts`：cooldown 初始化 + agent config merge 调用点（2-3 处重定位）
- 移除 0007 preroute（在 background model 下确实无意义）
- 移除 0004 shadow agent 注册（OpenCode 原生负责模型回退）

**第 3 步（1-2 天）：companion skill + 状态桥接**
- 创建 `background-orchestration/SKILL.md`：告知编排器异步调度语义
- BackgroundJobBoard 的 synthetic completion 到阶段状态机的桥接
- 可选状态持久化（文件级 JSON，防止 OpenCode 重启丢失）
- 轻量扩展 `verification-before-completion`（持久化验证条款）

**第 4 步（1 天）：端到端验证**
- brainstorm → plan → implement → review → verify → finish 全流程
- best-of-N 并行测试
- 状态恢复测试

### 为什么不是独立 adapter 架构（路径B）

简化版路径B在理论上仍然更"干净"，但修正后的事实（superpowers 无需改动、15 个策略文件零冲突）使得独立 adapter 层的成本-收益不再优：

- 额外抽象层增加调试复杂度
- 策略文件和 OMO Slim 仍然有紧密的运行时耦合（filter-available-skills hook 在 OMO Slim 内部）
- 1-1.5 周 vs 1-2 周的工作量差异不显著，但 adapter 方案增加了架构复杂度
